🔹 АРХИТЕКТУРА И КОД:
• Разделение ответственности: .razor — только разметка и привязка состояния. Бизнес-логика, парсинг, HTTP, БД — в отдельных сервисах/классах.
• Явная типизация: public API, параметры, возвращаемые значения, поля классов — только явные типы. var допускается только для очевидных локальных присваиваний.
• Порядок членов класса (строго): private readonly поля → public свойства → конструкторы (с null-проверками) → public методы → private хелперы → вложенные private классы.
• Blazor-правила: @code блоки тонкие. Использовать @inject, IDisposable для таймеров/подписок, CancellationToken для всех асинхронных операций. Избегать прямых StateHasChanged(); использовать InvokeAsync(StateHasChanged) при вызове из фоновых потоков.

🔹 ПРОТОКОЛ ИТЕРАЦИЙ:
• 1 этап → тест-чеклист → фикс ошибок → подтверждение стабилизации → следующий этап. Не генерировать несколько этапов за раз.
• Перед изменением DI, миграций БД, маршрутов, публичных API или структуры папок → всегда предупреждать и запрашивать явное подтверждение.
• Формат вывода: указывать точный путь к файлу, использовать diff-стиль или полный файл при правках. Если что-то неоднозначно или отсутствует в контексте → задавать уточняющий вопрос, а не гадать.
• Отказ от "магии": не использовать скрытые пакеты, не рекомендовать устаревшие API, не писать код, который нельзя протестировать изолированно.

🔹 БЕЗОПАСНОСТЬ И ПРОДАКШН-ГОТОВНОСТЬ:
• Секреты (API-ключи, строки подключения, пароли) → НИКОГДА не хардкодить. Только UserSecrets, env-переменные или Configuration.
• Валидация: все внешние данные (query-параметры, файлы, JSON, пользовательский ввод) → проверять перед использованием.
• Отмена операций: все async-методы принимают CancellationToken и корректно обрабатывают OperationCanceledException.
